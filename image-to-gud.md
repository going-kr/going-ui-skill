# 이미지 → .gud 변환 절차

> 레퍼런스 HMI 캡처 또는 스케치 이미지를 Going.UI .gud 파일로 변환할 때의 작업 절차와 주의사항.
> `gud-structure.md`, `controls/*.md` 등 JSON 문서와 함께 참조한다.

---

## 핵심 원칙

⚠ **원본 레이아웃 보존이 최우선이다.** 원본 이미지의 영역 구분, 비율, 시각적 구조를 .gud에 그대로 재현해야 한다. 임의 재구성은 절대 금지.

⚠ **Claude Vision은 정확한 픽셀 좌표를 반환할 수 없다.** 공간 관계(좌/우, 상/하, 비율)는 이해하지만, 정확한 px 좌표 추출은 불가. 따라서 **비율 기반으로 좌표를 계산**해야 한다:
- 원본 이미지에서 영역의 비율(%) 추정 → DesignWidth/Height에 적용 → Bounds 계산
- 예: "좌측 영역이 전체의 약 30%" → 1024 × 0.3 ≈ 307px

⚠ **단계적 분석이 필수.** 한 번에 전체 .gud를 생성하지 말 것. 아래 6단계를 순서대로 진행하라.

**원본 보존 세부 규칙:**
- 원본에서 GroupBox/테두리로 구분된 영역 → GoGroupBox 또는 GoPanel로 재현
- 원본의 열/행 비율 → GoTableLayoutPanel의 Columns/Rows 비율로 재현
- 원본에서 큰 영역(로그, 테이블)이 차지하는 면적비 → 동일 비율로 배치
- 원본에 없는 레이아웃 변경(예: 5열 균등 분할)을 임의로 적용하지 말 것
- 원본 재현과 개편은 반드시 분리 (아래 "개편이 포함된 경우" 섹션 참조)

---

## 절차 요약

```
1. 오브젝트 인벤토리   — 이미지의 모든 요소를 빠짐없이 식별
2. 그루핑             — 관련 오브젝트를 논리 그룹으로 묶기
3. 컨테이너 설계       — 그룹별 Going.UI 컨테이너 타입 결정 + 중첩 구조 설계
4. 원본 충실 도식화    — 이미지와 1:1 대응하는 레이아웃 도식 작성
5. 대조 검증          — 도식 vs 원본 이미지 비교, 누락·왜곡 수정
6. .gud JSON 생성     — 검증된 도식을 기반으로 .gud 파일 작성
```

---

## 1단계: 오브젝트 인벤토리

이미지에 보이는 **모든 시각 요소**를 빠짐없이 리스트업한다.

**식별 대상:**
- 데이터 표시: 테이블, 수치 표시, 그래프
- 상태 표시: 램프, 인디케이터, 아이콘
- 라벨: 헤더, 행/열 라벨, 그룹 타이틀, 섹션 타이틀
- 인터랙션: 버튼, 입력 필드, 슬라이더, 토글
- 장식: 배경 이미지, 구분선, 테두리

**주의사항:**
- 버튼은 특히 누락되기 쉽다 — 화면 하단/모서리의 작은 버튼, 탭 전환 버튼, 설정 버튼 등을 반드시 확인
- 테이블의 헤더 행과 데이터 행을 구분하여 카운트
- 라벨이 독립 요소인지 컨트롤의 부속(예: 램프 아래 텍스트)인지 구분

---

## 2단계: 그루핑

관련 오브젝트를 논리 그룹으로 묶는다. 각 그룹은 하나의 컨테이너 후보가 된다.

**그루핑 기준:**
- 기능적 연관성 (같은 데이터 영역, 같은 제어 대상)
- 공간적 인접성 (이미지에서 시각적으로 묶여 있는 것)
- 조작 빈도 (상시 감시 vs 가끔 조작)

**결정 사항:**
- 유사 그룹의 합치기/분리 여부 (예: 입력 램프와 출력 램프를 하나로? 별도로?)
- 장식 요소의 유지/제거 여부

---

## 3단계: 컨테이너 설계

각 그룹에 적합한 Going.UI 컨테이너를 선택하고 중첩 구조를 설계한다.

**컨테이너 분류:**

Going.UI 컨테이너는 **레이아웃**과 **섹션** 두 종류로 나뉜다.

*레이아웃 — 자식의 위치·크기를 결정하는 구조 컨테이너*

| 레이아웃 패턴 | 컨테이너 | 비고 |
|-------------|---------|------|
| 행×열 격자 배치 | GoTableLayoutPanel | 핵심 레이아웃. Columns/Rows로 비율 지정 |
| 탭 전환 영역 | GoTabControl | TabPosition으로 탭 위치 지정 |
| 페이지 전환 (탭 없이) | GoSwitchPanel | SetPage()로 코드 전환 |
| 스크롤 필요 영역 | GoScrollablePanel | 내용이 영역을 초과할 때 |

*섹션 — 논리적 그룹을 시각적으로 묶는 컨테이너*

GoPanel, GoBoxPanel, GoGroupBox는 모두 **섹션**이다. 역할은 동일하고 외형만 다르다.

| 컨테이너 | 외형 | 선택 기준 |
|---------|------|----------|
| GoPanel | 타이틀바 + 배경 + 테두리 | 그룹에 제목이 필요할 때 |
| GoBoxPanel | 배경 + 테두리 (타이틀 없음) | 제목 없이 영역만 구분할 때 |
| GoGroupBox | 테두리 + 라벨 텍스트 (배경 선택) | 테두리 위에 라벨을 올릴 때 |

**⚠️ 섹션 중첩 규칙 (필수):**

섹션 안에 섹션을 중첩할 때는 반드시 다음 규칙을 지킨다:

- **자식 섹션이 2개 이상** → 중첩 허용. 하나의 섹션이 여러 하위 섹션을 묶는 것은 의미가 있다.
- **자식 섹션이 1개뿐** → **모순**. 바깥 섹션이 존재할 이유가 없다. 둘 중 하나를 제거하거나 합칠 것.

```
✅ GoPanel "전체"
     ├─ GoGroupBox "센서"    ← 2개 이상이므로 중첩 OK
     └─ GoGroupBox "출력"

❌ GoPanel "전체"
     └─ GoGroupBox "센서"    ← 1개뿐 → 모순. GoPanel 하나로 합칠 것
```

**중첩 설계 원칙:**
- 최상위는 GoTableLayoutPanel로 화면을 대영역으로 분할
- 각 대영역 안에서 섹션(GoPanel/GoBoxPanel/GoGroupBox)으로 그룹 시각화
- 섹션 안에서 자식 배치가 필요하면 GoTableLayoutPanel 등 레이아웃을 중첩
- 섹션 → 단일 섹션 중첩은 금지

---

## 4단계: 원본 충실 도식화 ★

> **이 단계가 핵심이다. 생략하면 누락·왜곡이 발생한다.**

설계한 컨테이너 구조를 **원본 이미지와 1:1로 대응**시켜 도식화한다.

**필수 확인 항목:**

| 항목 | 확인 내용 |
|------|----------|
| 공간 점유 | 원본에서 꽉 차 있는 영역이 도식에서도 꽉 차는가 |
| 행/열 존재 | 원본의 모든 독립 행·열이 도식에 반영되었는가 |
| 버튼/컨트롤 위치 | 원본에서 별도 행·영역을 차지하는 버튼이 도식에서도 독립 영역인가 |
| 비율 | 원본의 대략적인 면적 비율이 도식에 반영되었는가 |
| 빈 공간 | 원본에 없는 빈 공간이 도식에 생기지 않았는가 |

**흔한 실수:**
- 버튼을 인벤토리에서 식별했지만 컨테이너 배치에서 독립 Row로 잡지 않음
- 좌측 영역이 원본에서 세로 전체를 채우는데 도식에서 내용물 높이만큼만 차지함
- 개편 의도를 반영하면서 원본 구조를 왜곡 (개편은 6단계 이후 별도 진행)

---

## 5단계: 대조 검증

도식과 원본 이미지를 나란히 놓고 비교한다.

**검증 체크리스트:**

- [ ] 인벤토리의 모든 오브젝트가 도식에 존재하는가
- [ ] 각 컨테이너의 자식 컨트롤 수가 원본과 일치하는가
- [ ] 원본의 공간 비율(좌:우, 상:하)이 도식에 반영되었는가
- [ ] 원본에서 독립 영역인 요소(버튼 행, 타이틀바 등)가 도식에서도 독립 영역인가
- [ ] 원본에 없는 빈 공간이 도식에 존재하지 않는가
- [ ] 컨테이너 타입이 원본의 레이아웃 패턴과 맞는가

**불일치 발견 시:** 3단계로 돌아가 컨테이너 구조를 수정한 뒤 다시 4~5단계 반복.

---

## 6단계: .gud JSON 생성

검증된 도식을 기반으로 .gud 파일을 작성한다.

**작업 순서:**
1. GoDesign 구조 생성 (Pages, Windows)
2. 최상위 컨테이너부터 바깥→안쪽 순서로 JSON 작성
3. 각 컨트롤에 고유 UUID Id 부여
4. Bounds를 `"L,T,R,B"` 형식으로 설정 (컨테이너 내부 배치 시에는 Dock 활용)
5. Name, DesignWidth, DesignHeight 설정 (단일 GoDesign JSON)

**JSON 작성 시 참조:**
- .gud 최상위 구조 → `gud-structure.md`
- 컨트롤별 JSON → `controls/*.md`
- 데이터 클래스 (GoListItem 등) → `controls/_common.md`
- Enum 숫자값 → `enums-reference.md`

---

## 개편이 포함된 경우

원본 재현과 개편은 **반드시 분리**하여 진행한다.

```
[1~5단계] 원본 충실 재현 → 검증 완료
    ↓
[개편 설계] 변경할 내용 정리 (레이아웃 변경, 컨트롤 추가/제거, 위계 변경)
    ↓
[4~5단계 반복] 개편안 도식화 → 대조 검증 (개편 의도 vs 도식)
    ↓
[6단계] 개편안 기준으로 .gud 생성
```

원본 재현 없이 바로 개편안으로 .gud를 작성하면, 원본의 요소가 누락되거나 공간 구조가 왜곡되는 문제가 발생한다.

---

## 좌표 정밀도 검증 체크리스트

이미지에서 추출한 좌표를 .gud JSON으로 변환할 때 아래 체크리스트로 정밀도를 검증한다.

### 비율 검증
- [ ] DesignWidth/DesignHeight가 원본 이미지의 가로:세로 비율과 일치하는가?
- [ ] 각 컨트롤의 Bounds 좌표가 원본 이미지 내 위치와 ±5px 이내로 일치하는가?
- [ ] 컨테이너(GoTableLayoutPanel 등)의 Columns/Rows 비율이 이미지의 분할 비율과 일치하는가?

### 크기 검증
- [ ] 버튼/입력 컨트롤의 최소 크기가 확보되었는가? (GoButton 최소 60×30, GoSlider 최소 150×50)
- [ ] 텍스트가 잘리지 않을 만큼 컨트롤 크기가 충분한가?
- [ ] Margin(기본 3px)을 감안한 실제 표시 영역이 의도와 일치하는가?

### 배치 검증
- [ ] 겹치는 컨트롤이 없는가? (Bounds 영역이 다른 컨트롤과 교차하지 않는가?)
- [ ] Dock=Fill인 컨트롤이 컨테이너 전체를 정확히 채우는가?
- [ ] GoTableLayoutPanel의 ColSpan/RowSpan이 이미지의 셀 병합과 일치하는가?

### 최종 대조
- [ ] .gud를 UIEditor에서 열었을 때 원본 이미지와 시각적으로 일치하는가?
- [ ] 오차가 ±5px를 초과하는 컨트롤이 있으면 사용자에게 보고하고 수동 조정 안내
