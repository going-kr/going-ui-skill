# Going UI — 세션 간 컨텍스트 유지

> **이 문서는 Claude Code 환경에서만 적용된다.** 채팅 환경에서는 파일 시스템 접근이 불가능하므로 해당 없음.

Going Library 프로젝트는 여러 세션에 걸쳐 작업하는 경우가 많다. 새 세션에서 이전 작업을 이어받으려면 **프로젝트 상태 파일**을 유지해야 한다.

---

## 세션 전환 시 판단 흐름

새 세션이 시작되면 아래 순서로 현재 상태를 파악한다.

```
1. project_status.md 존재?
   ├─ YES → Read로 읽고 현재 단계 파악
   │   ├─ 이전 세션이 .gud 생성까지 완료 → Step 3(UIEditor) 완료 여부 확인
   │   │   ├─ Designer.cs 존재 → Step 4(C# 코드) 진행
   │   │   └─ Designer.cs 없음 → "MakeCode 실행 후 알려주세요" 안내
   │   ├─ 이전 세션이 C# 코드까지 완료 → 빌드/배포 진행
   │   └─ 이전 세션이 project_brief만 완료 → .gud 생성 진행
   └─ NO → 새 프로젝트 시작 (project-brief.md 인터뷰)
```

**핵심 판단 기준:**
- `design.json` + `*.Designer.cs` 존재 → MakeCode 완료, C# 작업 가능
- `.gud` 존재 + `Designer.cs` 없음 → UIEditor + MakeCode 필요 (사용자 액션)
- `*.cs` 사용자 파일에 코드가 있음 → C# 작업 진행 중 또는 완료
- `project_brief.md`만 존재 → .gud 생성부터 시작

**채팅 vs Claude Code 분기:**

| 상태 | 채팅 환경 | Claude Code |
|------|----------|-------------|
| brief만 있음 | .gud JSON 생성 가능 | .gud 파일 생성 가능 |
| .gud까지 완료 | 여기까지. UIEditor 안내 | UIEditor 안내 |
| MakeCode 완료 | C# 코드 작성 불가 | C# 코드 작성 진행 |
| C# 완료 | 빌드 불가 | `dotnet build` 실행 |

## 프로젝트 상태 파일 (`project_status.md`)

프로젝트 루트에 `project_status.md` 파일을 생성하여 아래 내용을 유지한다.
**세션 시작 시 이 파일이 존재하면 반드시 읽고 현재 상태를 파악한 후 작업을 시작할 것.**

```markdown
## 프로젝트 개요
- 프로젝트명, 해상도, 통신 방식 등 핵심 정보 요약

## 현재 단계
- [ ] 1. project_brief.md 작성
- [ ] 2. .gud 파일 생성
- [ ] 3. UIEditor + MakeCode (사용자)
- [ ] 4. C# 코드 구현
- [ ] 5. 빌드 및 실행
- [ ] 6. 장치 배포

## 완료된 작업
- [x] 완료된 항목 (날짜)

## 진행 중인 작업
- [ ] 현재 작업 중인 항목

## 이슈 사항
- 알려진 버그, 보류 사항, 사용자 확인 대기 항목

## 작업 노트
### 2026-03-21
- 작업 내용 기록
```

## 유지 규칙

1. **작업 시작 시** — `project_status.md`가 있으면 `Read`로 읽고 현재 단계 파악. 위 판단 흐름에 따라 다음 작업 결정
2. **단계 완료 시** — 현재 단계 체크박스 업데이트, 완료 항목에 날짜와 함께 기록
3. **이슈 발생 시** — 이슈 사항에 기록 (사용자 확인 대기, 알려진 버그 등)
4. **세션 종료 시** — 작업 노트에 해당 날짜의 작업 내용 기록. 다음 세션에서 이어받을 수 있도록 현재 상태를 명확히 남길 것

> 이 파일은 Claude Code가 자동으로 유지한다. 사용자가 별도로 관리할 필요 없음.
