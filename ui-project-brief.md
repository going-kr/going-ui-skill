# project_brief.md 작성 가이드

## 파일 구조

project_brief.md 위치: 프로젝트 루트/project_brief.md
이미지 위치:           프로젝트 루트/images/

이미지 네이밍 컨벤션:
  page_{페이지명소문자}.jpg   ← 페이지 레이아웃 (스케치 또는 레퍼런스 HMI 캡처)
  wnd_{윈도우명소문자}.jpg    ← 윈도우 레이아웃
  ref_{설명}.jpg             ← 기타 레퍼런스

Claude에게 전달 시: project_brief.md 텍스트 + images/ 폴더의 이미지를 채팅에 함께 첨부.

### 샘플 프로젝트 구조

`/going-ui-skill/sample/` 에 실제 프로젝트 예시가 있음. 새 프로젝트 작성 시 참고:

```
sample/
├── project_brief.md         ← 프로젝트 브리프 (완성 예시)
├── images/                  ← 레이아웃 이미지 (Claude에게 전달하는 이미지)
│   ├── page_monitoring.png
│   ├── page_monitoring_layout.png   ← 레이아웃 구조도
│   ├── wnd_calibration.png
│   └── wnd_calibration_layout.png   ← 레이아웃 구조도
├── reference/               ← 레퍼런스 원본 (기존 HMI 캡처, 데이터시트 등)
│   ├── Monitoring.png       ← 원본 HMI 화면 캡처
│   ├── CalibrationWindow.png
│   └── *.xlsx               ← 레지스터 맵 원본 등
└── ui/                      ← 생성된 .gud 파일
    └── ProjectName.gud
```

> `reference/` = 원본 자료 보관용, `images/` = 레퍼런스를 기반으로 작성된 레이아웃 이미지

### 샘플 참조 방법

새 프로젝트 작성 시 `sample/` 폴더를 다음과 같이 활용:

1. **project_brief.md 작성법 참고** — `sample/project_brief.md`는 완성된 프로젝트 브리프 예시. 프로젝트 개요, 화면 구성(페이지·윈도우 목록), 통신(Modbus 레지스터 맵), 설정 파일, 배포 정보가 모두 포함된 실제 작업 사례
2. **레퍼런스 → 레이아웃 이미지 변환 과정 참고** — `reference/Monitoring.png`(원본 HMI)와 `images/page_monitoring_layout.png`(레이아웃 구조도)를 비교하면 원본에서 레이아웃 이미지를 어떻게 작성하는지 이해할 수 있음
3. **최종 .gud 결과물 참고** — `ui/SenvasSample.gud`는 위 브리프+이미지를 기반으로 생성된 실제 .gud 파일. Pages/Windows Dictionary 구조, GoTableLayoutPanel `"%"` 레이아웃, 컨트롤 배치 등의 실제 결과를 확인 가능

> **핵심**: 새 프로젝트 시 `sample/project_brief.md` 구조를 따라 작성하고, `sample/ui/*.gud`의 JSON 구조를 참고하여 .gud를 생성할 것

## 이미지 종류와 워크플로우

### 이미지 분류

| 이미지 종류 | 접두사 | 설명 |
|-----------|--------|------|
| 레퍼런스 이미지 | `ref_` | 원본 이미지 (기존 HMI 캡처, 디자인 시안, 타사 화면 등). `reference/` 폴더에 저장 |
| 레이아웃 이미지 | `page_*_layout`, `wnd_*_layout` | 레퍼런스 이미지를 기반으로 작성된 레이아웃 구조도. `images/` 폴더에 저장 |
| 페이지/윈도우 이미지 | `page_*`, `wnd_*` | 레이아웃 이미지가 없을 때 직접 사용하는 스케치 또는 레퍼런스 |

### .gud 생성 워크플로우

1. **레이아웃 이미지가 별도로 있는 경우** (권장)
   - 레이아웃 이미지(`*_layout.*`)를 기준으로 전체적 레이아웃 구조를 먼저 설정
   - 레퍼런스 이미지(`ref_*`)로 세부 컨트롤 배치·스타일을 확인
   - 레이아웃 이미지 → 구조(TableLayout/컨테이너), 레퍼런스 이미지 → 디테일(컨트롤 종류·값·색상)

2. **레이아웃 이미지가 없는 경우**
   - 레퍼런스 이미지를 분석하여 전체적 레이아웃을 먼저 추론
   - 큰 영역(헤더, 본문 좌/우, 하단 등)을 TableLayout `"%"` 비율로 나눔
   - 그 안에 세부적 레이아웃(컨트롤 그룹별 컨테이너)을 잡은 후 컨트롤 배치

3. **이미지가 전혀 없는 경우**
   - project_brief.md 텍스트 설명만으로 레이아웃을 설계
   - TableLayout `"%"` 비율 기반 반응형 레이아웃 우선

> **핵심 원칙**: 항상 "전체 레이아웃 → 세부 레이아웃 → 컨트롤 배치" 순서로 진행. TableLayout Columns/Rows는 `"%"` 비율을 기본으로 사용.

## 템플릿

> **아래 구분선부터 끝까지가 복사할 템플릿.** 새 프로젝트 시작 시 아래 내용을 `project_brief.md`로 복사한 뒤 각 항목을 채워넣을 것.

### 완성도 체크리스트

작성 후 아래 항목이 모두 충족되었는지 확인:

- [ ] **프로젝트명**이 유효한 C# 식별자인가 (영문+숫자+밑줄, 숫자 시작 불가)
- [ ] **페이지/윈도우 목록**이 빠짐없이 작성되었는가
- [ ] **레이아웃 이미지** (스케치 또는 레퍼런스)가 각 페이지/윈도우마다 있는가
- [ ] **페이지 간 이동** 관계가 명시되었는가 (어떤 버튼 → 어느 페이지)
- [ ] **통신 방식**이 명시되었는가 (Modbus/MQTT/없음)
- [ ] Modbus 사용 시 **레지스터 맵**이 작성되었는가 (주소, 명칭, 단위, R/W, FC)
- [ ] Modbus 사용 시 **슬레이브 구성**이 작성되었는가
- [ ] **설정 파일 항목**이 정리되었는가

> 레지스터 맵 누락 시 Claude Code가 DeviceManager를 작성할 수 없음 — 가장 흔한 병목.

### 통신 섹션 필수 검증 (AI 1차 검증)

project_brief.md 작성 완료 후, 통신 섹션에 대해 아래 항목을 **반드시 검증**한다.
누락 항목이 있으면 사용자에게 "아래 항목이 빠져 있습니다"라고 안내하고, **누락 상태에서 통신 코드 작성을 진행하지 말 것.**

> **"통신 없음"으로 명시된 경우** 이 검증을 건너뛴다. 단, 화면에 통신 관련 컨트롤(GoLamp 통신상태, GoValue 센서값 등)이 있는데 "통신 없음"이면 모순이므로 사용자에게 확인할 것.

#### Modbus RTU 사용 시

- [ ] **포트명**이 명시되었는가 (COM1, /dev/ttyUSB0 등)
- [ ] **보드레이트**가 명시되었는가 (9600, 19200, 115200 등)
- [ ] **패리티**가 명시되었는가 (None, Even, Odd)
- [ ] **타임아웃**이 명시되었는가
- [ ] **슬레이브 번호**와 **역할**이 명시되었는가
- [ ] **레지스터 맵**이 작성되었는가 — 아래 항목 모두 포함:
  - [ ] 시작 주소
  - [ ] 각 주소의 데이터 명칭
  - [ ] 단위 (℃, %, rpm 등 — 스케일링 필요 시 배율도)
  - [ ] R/W 구분
  - [ ] FC 코드 (FC3/FC1/FC6/FC16 등)
- [ ] **비트 맵**이 필요한 경우 작성되었는가 (FC1/FC5 영역)

#### Modbus TCP 사용 시

- [ ] **IP 주소**가 명시되었는가
- [ ] **포트 번호**가 명시되었는가 (기본 502)
- [ ] 나머지는 RTU와 동일 항목 검증

#### MQTT 사용 시

- [ ] **브로커 주소**가 명시되었는가
- [ ] **포트 번호**가 명시되었는가 (기본 1883)
- [ ] **Subscribe 토픽** 목록이 있는가 (토픽명 + 페이로드 형식)
- [ ] **Publish 토픽** 목록이 있는가 (토픽명 + 페이로드 형식)
- [ ] 페이로드가 JSON인 경우 **필드 구조**가 명시되었는가

#### CNet (LS Electric) 사용 시

- [ ] **포트명/보드레이트**가 명시되었는가
- [ ] **디바이스 주소**가 LS 주소체계(%MW, %MB 등)로 작성되었는가
- [ ] **읽기 영역** (시작 주소 + 길이)이 명시되었는가
- [ ] **쓰기 주소**가 명시되었는가

#### MC (Mitsubishi) 사용 시

- [ ] **IP/포트**가 명시되었는가
- [ ] **디바이스 주소**가 Mitsubishi 주소체계(D0, M0 등)로 작성되었는가
- [ ] **읽기 영역** (시작 주소 + 길이)이 명시되었는가
- [ ] **쓰기 주소**가 명시되었는가

> **Why:** 통신 파라미터가 하나라도 빠지면 DeviceManager 코드를 정확히 작성할 수 없다. 특히 레지스터 맵 누락은 가장 빈번한 병목이며, 잘못된 주소로 쓰기를 실행하면 산업 장비 오동작으로 이어질 수 있다.

## 3. 통신

### Modbus RTU (해당 없으면 섹션 삭제)

- **포트명**: COM1 / /dev/ttyUSB0 등
- **보드레이트**: 9600 / 19200 / 115200
- **패리티**: None / Even / Odd
- **타임아웃(ms)**: 1000

**슬레이브 구성**

| 슬레이브 번호 | 역할 |
|-------------|------|
| 1 | 메인 컨트롤러 |

**레지스터 맵 (워드)**

| 주소 | 명칭 | 설명 | 단위 | R/W | FC |
|------|------|------|------|-----|-----|
| 0x0000 | RunState | 운전 상태 (0=정지, 1=운전) | - | R | FC3 |
| 0x0001 | ErrorCode | 에러 코드 | - | R | FC3 |
| 0x000A | Temperature | 온도 | 0.1℃ | R | FC3 |
| 0x0064 | SetPoint | 설정값 | - | R/W | FC3/FC6 |

**비트 맵 (해당 시)**

| 주소 | 명칭 | 설명 | R/W | FC |
|------|------|------|-----|-----|
| 0x0000 | RunCmd | 운전 명령 | R/W | FC1/FC5 |

### Modbus TCP (해당 없으면 섹션 삭제)

- **IP**: 192.168.0.xxx
- **포트**: 502
- **타임아웃(ms)**: 1000

슬레이브 구성 및 레지스터 맵은 RTU와 동일 형식으로 작성.

### MQTT (해당 없으면 섹션 삭제)

- **브로커 주소**: 192.168.0.xxx
- **포트**: 1883

| 방향 | 토픽 | 페이로드 형식 | 설명 |
|------|------|------------|------|
| Subscribe | sensor/temperature | {"value": 25.3} | 온도 수신 |
| Publish | command/start | "1" | 운전 명령 |
