# project_brief.md 작성 가이드

## 인터뷰 절차

project_brief.md 작성은 3가지 경로 중 하나로 진행한다.

| 경로 | 조건 | 설명 |
|------|------|------|
| A | 프로젝트 구상만 있고 자료가 없을 때 | Claude와 인터뷰로 brief 작성 |
| B | 사용자가 brief 템플릿을 직접 채울 때 | 템플릿 제공 → 검증 |
| C | 기존 자료(이미지, 엑셀 등)가 이미 있을 때 | 자료 분석 → 빠진 부분만 인터뷰 → brief 생성 |

---

### 경로 A — Claude와 인터뷰를 통해 작성

인터뷰는 3단계로 진행한다. 각 단계는 독립적으로 실행 가능하며,
단계 완료 시 결과를 요약하여 사용자 확인을 받은 뒤 다음으로 넘어간다.
(절대 규칙 #8 적용)

#### 1단계: 프로젝트 윤곽 (발산)

목적: 프로젝트의 전체 그림을 빠르게 파악한다.
사용자가 이미 알고 있는 정보를 최대한 끌어낸다.

**질문 항목:**
1. 프로젝트명, 앱 이름, 목적, 해상도, 테마
2. 페이지 목록 (각 페이지 역할, 대략적인 화면 구성)
3. 윈도우(팝업) 목록 (트리거, 역할)
4. 통신 방식 선택 (Modbus RTU/TCP, MQTT, 없음)
5. 기존 HMI/레퍼런스 이미지가 있는지

**경로 전환:** ⑤에서 사용자가 기존 자료(이미지, 엑셀 등)를 폴더 또는 파일로 제공한 경우, 경로 C의 0단계로 전환한다. 이미 파악된 ①~④ 정보는 유지하고 중복 질문하지 않는다.

**진행 방법:**
- 한 번에 하나씩 질문한다. 복잡한 항목(통신, 레지스터 맵)은 반드시 개별 질문. 간단한 항목(프로젝트명+해상도+테마)은 2~3개 묶어도 된다.
- 선택지가 있는 항목은 객관식으로 제시한다.
  - 예: "통신 방식이 뭔가요?" (X) → "Modbus RTU / Modbus TCP / MQTT / 없음 중 어떤 건가요?" (O)
- 사용자 답변에서 언급되지 않은 부분을 역으로 짚어준다.
  - 예: "페이지 3개를 말씀하셨는데, 설정 페이지는 별도로 필요하지 않으신가요?"
- 모호한 답변은 구체화 질문으로 확인한다.
  - 예: "모니터링 페이지에 온도를 표시한다고 하셨는데, 그래프로 보여줄지 숫자만 표시할지 어떤 형태인가요?"
- 지금 당장 필요하지 않은 기능은 제외를 권한다. (YAGNI)
  - 예: "이 기능은 나중에 추가할 수 있으니 1차에서는 빼는 게 어떨까요?"
- 구조 결정 시 2~3가지 대안을 먼저 제시하고 사용자가 선택하게 한다.
  - 예: "모니터링과 트렌드를 한 페이지에 탭으로 넣는 방법(A)과 별도 페이지로 분리하는 방법(B)이 있는데, 어떤 게 맞나요?"

**완료 조건:**
- 페이지/윈도우 목록이 확정됨
- 통신 방식이 결정됨
- 결과를 요약하여 사용자에게 보여주고 확인받음

#### 2단계: 상세 설계 (수렴)

목적: 1단계에서 확정된 윤곽을 구현 가능한 수준으로 구체화한다.
한 번에 전부 보여주지 않고 청크 단위로 끊어서 확인받는다.

**청크 A — 화면별 상세:**
- 페이지별: 주요 컨트롤 목록, 페이지 간 이동 관계, 열리는 윈도우
- 윈도우별: 반환값, 내부 컨트롤
- 레이아웃 이미지가 있으면 이미지 기준으로 확인
- 빠진 시나리오 탐색: "에러 발생 시 화면 처리는?", "통신 끊김 시 UI 동작은?"
- → 사용자 확인

**청크 B — 통신 상세** (1단계에서 "통신 없음" 선택 시 건너뛴다):
- Modbus: 포트, 보드레이트, 슬레이브 구성
- 레지스터 맵 (주소, 명칭, 단위, R/W, FC)
- MQTT: 브로커, 토픽/페이로드 구조
- ⚠ 레지스터 맵은 사용자만 알고 있는 정보 — 반드시 사용자가 제공해야 함. 추측하여 작성하지 않는다. (절대 규칙 #4)
- → 사용자 확인

**청크 C — 설정 및 배포:**
- 설정 파일 항목 (DataManager에 저장할 값)
- 배포 대상 장치, 자동실행/키오스크 여부
- → 사용자 확인

**완료 조건:**
- 모든 청크가 사용자 확인을 받음
- 결정 사항에 대한 이유가 명확함 (왜 이 통신 방식인지, 왜 이 페이지 구조인지)

#### 3단계: brief 생성 및 검증

목적: 1~2단계 결과를 project_brief.md로 조립하고 최종 검증한다.

- project_brief.md 초안 생성 (아래 템플릿 기준)
- 완성도 체크리스트를 항목별로 대조하며 결과를 명시한다. "통과"라고만 말하지 않고 각 항목의 실제 내용을 보여주며 확인:
  - □ 프로젝트명이 유효한 C# 식별자인가 → "SenvasMonitor" ✅
  - □ 페이지/윈도우 목록이 빠짐없는가 → "3 페이지, 2 윈도우" ✅
  - □ 레이아웃 이미지가 각 화면마다 있는가 → "MainPage 없음" ❌ → 요청
  - □ 페이지 간 이동 관계가 명시되었는가 → "Main↔Setting, Main↔Alarm" ✅
  - □ 통신 방식 + 레지스터 맵이 완비되었는가 → "RTU, 32워드, 8비트" ✅
  - □ 설정 파일 항목이 정리되었는가 → "PortName, Baudrate, Timeout" ✅
- 누락 항목이 있으면 해당 청크로 돌아가 보완
- 사용자 최종 승인

**완료 조건:**
- project_brief.md가 완성됨
- 체크리스트 전 항목 통과 — 항목별 증거 명시
- 사용자가 "이대로 진행" 확인

#### brief 완성 후 파일 맵핑

brief가 확정되면, 구현 단계에 앞서 생성/수정될 파일 목록을 정리한다.
사용자에게 보여주고 확인받은 뒤 다음 단계(.gud 생성)로 넘어간다.

> 예:
> "이 brief로 생성될 파일:
>  - SenvasMonitor.gud (3페이지, 2윈도우)
>  - Pages/MainPage.cs, SettingPage.cs, AlarmPage.cs
>  - Windows/CalibrationWindow.cs, KeypadWindow.cs
>  - DeviceManager.cs (MasterRTU, 슬레이브 2개)
>  - DataManager.cs (설정 3항목)
>  이 구조로 진행할까요?"

### 경로 B — 템플릿 직접 작성

사용자가 project_brief.md 템플릿을 직접 채우는 경우.

1. 사용자가 템플릿을 요청하면:
   - Claude Code: 아래 템플릿 섹션을 프로젝트 루트에 `project_brief.md`로 복사하여 제공
   - 채팅: 템플릿 내용을 텍스트로 출력

2. 사용자가 작성한 brief를 제출하면:
   - 3단계(검증) 체크리스트로 검증
   - 누락 항목이 있으면 해당 부분만 질문하여 보완
   - 체크리스트 전 항목 통과 후 파일 맵핑 → 사용자 확인

### 경로 C — 기존 자료 기반 brief 생성

**트리거:** 사용자가 프로젝트 자료(이미지, 엑셀, 문서 등)를 폴더 경로 지정 또는 파일 첨부로 제공하며 개발을 요청한 경우.
예: "이 폴더 보고 개발 시작해줘", "여기 자료 넣어놨어", (파일을 채팅에 직접 첨부)

#### 0단계: 제공된 자료 분석

목적: 사용자가 제공한 자료에서 brief에 필요한 정보를 최대한 추출한다.

1. 지정된 경로 또는 첨부 파일 전체 스캔
   - 이미지 파일(png/jpg/bmp) → 화면 후보 식별
   - 스프레드시트(xlsx/csv) → 레지스터 맵/데이터 구조 파싱
   - 문서(pdf/docx/txt/md) → 프로젝트 맥락 파악
   - 기타 파일 → 용도 판단 보류, 사용자에게 확인

   > ⚠ 채팅 환경에서는 xlsx 파일을 직접 파싱할 수 없다. 사용자에게 레지스터 맵을 텍스트/표로 붙여달라고 요청하거나, Claude Code 환경에서 진행하도록 안내한다.

2. 분석 결과를 파일 목록 + 용도 추정으로 요약하여 보고 → 사용자 확인

> 예:
> "제공된 자료를 분석했습니다:
>  - 화면 이미지 3장: Monitoring.png, Setting.png, Alarm.png
>  - 엑셀 1개: register_map.xlsx (슬레이브 2개, 워드 32개, 비트 8개)
>  - 용도 불명 1개: spec_v2.pdf
>  이 파악이 맞나요? spec_v2.pdf는 어떤 내용인가요?"

#### 1단계: 발산 (빠진 부분만 질문)

**핵심 원칙:** 자료에서 이미 파악된 정보는 다시 묻지 않는다. brief에 필요하지만 자료에 없는 항목만 질문한다.

- 이미지에서 페이지 3개는 파악했지만 페이지 간 이동이 없으면: "Monitoring이 메인이고, 설정 버튼 → Setting, 알람 버튼 → Alarm 이런 구조인가요?"
- 엑셀에 통신 파라미터(포트, 보드레이트)가 없으면: "Modbus RTU 포트는 /dev/ttyUSB0, 보드레이트 9600으로 할까요?"
- 프로젝트명, 앱 이름, 해상도 등 기본 정보: "프로젝트명은 뭘로 할까요? 해상도는 1024x600 기본으로 갈까요?"

진행 방법은 경로 A 1단계와 동일 (객관식 우선, 하나씩 질문, YAGNI, 대안 제시)

**완료 조건:** 경로 A 1단계와 동일

#### 2단계: 수렴 (자료 기반 청크 확인)

자료에서 추출한 정보를 brief 형태로 정리하여 청크별로 확인받는다.

**청크 A — 이미지 기반 화면 구조:**
이미지를 분석하여 페이지별 컨트롤 목록, 레이아웃 구조를 정리 → 사용자 확인

**청크 B — 엑셀 기반 통신/레지스터 맵:**
엑셀 데이터를 그대로 brief 레지스터 맵 형식으로 변환하여 확인 → 사용자 확인

**청크 C — 설정/배포 (자료에 없으므로 질문):**
경로 A 2단계 청크 C와 동일 → 사용자 확인

**완료 조건:** 경로 A 2단계와 동일

#### 3단계: brief 생성 및 검증

경로 A 3단계와 동일. 체크리스트 항목별 증거 명시 + 파일 맵핑.

## 파일 구조

project_brief.md 위치: 프로젝트 루트/project_brief.md
이미지 위치:           프로젝트 루트/images/

이미지 네이밍 컨벤션:
  page_{페이지명소문자}.jpg   ← 페이지 레이아웃 (스케치 또는 레퍼런스 HMI 캡처)
  wnd_{윈도우명소문자}.jpg    ← 윈도우 레이아웃
  ref_{설명}.jpg             ← 기타 레퍼런스

Claude에게 전달 시: project_brief.md 텍스트 + images/ 폴더의 이미지를 채팅에 함께 첨부.

### 샘플 프로젝트 구조

`/going-ui-skill/sample/` 에 실제 프로젝트 예시가 있음. 새 프로젝트 작성 시 참고:

```
sample/
├── project_brief.md         ← 프로젝트 브리프 (완성 예시)
├── images/                  ← 레이아웃 이미지 (Claude에게 전달하는 이미지)
│   ├── page_monitoring.png
│   ├── page_monitoring_layout.png   ← 레이아웃 구조도
│   ├── wnd_calibration.png
│   └── wnd_calibration_layout.png   ← 레이아웃 구조도
├── reference/               ← 레퍼런스 원본 (기존 HMI 캡처, 데이터시트 등)
│   ├── Monitoring.png       ← 원본 HMI 화면 캡처
│   ├── CalibrationWindow.png
│   └── *.xlsx               ← 레지스터 맵 원본 등
└── ui/                      ← 생성된 .gud 파일
    └── ProjectName.gud
```

> `reference/` = 원본 자료 보관용, `images/` = 레퍼런스를 기반으로 작성된 레이아웃 이미지

### 샘플 참조 방법

새 프로젝트 작성 시 `sample/` 폴더를 다음과 같이 활용:

1. **project_brief.md 작성법 참고** — `sample/project_brief.md`는 완성된 프로젝트 브리프 예시. 프로젝트 개요, 화면 구성(페이지·윈도우 목록), 통신(Modbus 레지스터 맵), 설정 파일, 배포 정보가 모두 포함된 실제 작업 사례
2. **레퍼런스 → 레이아웃 이미지 변환 과정 참고** — `reference/Monitoring.png`(원본 HMI)와 `images/page_monitoring_layout.png`(레이아웃 구조도)를 비교하면 원본에서 레이아웃 이미지를 어떻게 작성하는지 이해할 수 있음
3. **최종 .gud 결과물 참고** — `ui/SenvasSample.gud`는 위 브리프+이미지를 기반으로 생성된 실제 .gud 파일. Pages/Windows Dictionary 구조, GoTableLayoutPanel `"%"` 레이아웃, 컨트롤 배치 등의 실제 결과를 확인 가능

> **핵심**: 새 프로젝트 시 `sample/project_brief.md` 구조를 따라 작성하고, `sample/ui/*.gud`의 JSON 구조를 참고하여 .gud를 생성할 것

## 이미지 종류와 워크플로우

### 이미지 분류

| 이미지 종류 | 접두사 | 설명 |
|-----------|--------|------|
| 레퍼런스 이미지 | `ref_` | 원본 이미지 (기존 HMI 캡처, 디자인 시안, 타사 화면 등). `reference/` 폴더에 저장 |
| 레이아웃 이미지 | `page_*_layout`, `wnd_*_layout` | 레퍼런스 이미지를 기반으로 작성된 레이아웃 구조도. `images/` 폴더에 저장 |
| 페이지/윈도우 이미지 | `page_*`, `wnd_*` | 레이아웃 이미지가 없을 때 직접 사용하는 스케치 또는 레퍼런스 |

### .gud 생성 워크플로우

1. **레이아웃 이미지가 별도로 있는 경우** (권장)
   - 레이아웃 이미지(`*_layout.*`)를 기준으로 전체적 레이아웃 구조를 먼저 설정
   - 레퍼런스 이미지(`ref_*`)로 세부 컨트롤 배치·스타일을 확인
   - 레이아웃 이미지 → 구조(TableLayout/컨테이너), 레퍼런스 이미지 → 디테일(컨트롤 종류·값·색상)

2. **레이아웃 이미지가 없는 경우**
   - 레퍼런스 이미지를 분석하여 전체적 레이아웃을 먼저 추론
   - 큰 영역(헤더, 본문 좌/우, 하단 등)을 TableLayout `"%"` 비율로 나눔
   - 그 안에 세부적 레이아웃(컨트롤 그룹별 컨테이너)을 잡은 후 컨트롤 배치

3. **이미지가 전혀 없는 경우**
   - project_brief.md 텍스트 설명만으로 레이아웃을 설계
   - TableLayout `"%"` 비율 기반 반응형 레이아웃 우선

> **핵심 원칙**: 항상 "전체 레이아웃 → 세부 레이아웃 → 컨트롤 배치" 순서로 진행. TableLayout Columns/Rows는 `"%"` 비율을 기본으로 사용.

project_brief.md 작성 완료 후, 통신 섹션에 대해 아래 항목을 **반드시 검증**한다.
누락 항목이 있으면 사용자에게 "아래 항목이 빠져 있습니다"라고 안내하고, **누락 상태에서 통신 코드 작성을 진행하지 말 것.**

#### Modbus RTU 사용 시

- [ ] **포트명**이 명시되었는가 (COM1, /dev/ttyUSB0 등)
- [ ] **보드레이트**가 명시되었는가 (9600, 19200, 115200 등)
- [ ] **패리티**가 명시되었는가 (None, Even, Odd)
- [ ] **타임아웃**이 명시되었는가
- [ ] **슬레이브 번호**와 **역할**이 명시되었는가
- [ ] **레지스터 맵**이 작성되었는가 — 아래 항목 모두 포함:
  - [ ] 시작 주소
  - [ ] 각 주소의 데이터 명칭
  - [ ] 단위 (℃, %, rpm 등 — 스케일링 필요 시 배율도)
  - [ ] R/W 구분
  - [ ] FC 코드 (FC3/FC1/FC6/FC16 등)
- [ ] **비트 맵**이 필요한 경우 작성되었는가 (FC1/FC5 영역)

#### Modbus TCP 사용 시

- [ ] **IP 주소**가 명시되었는가
- [ ] **포트 번호**가 명시되었는가 (기본 502)
- [ ] 나머지는 RTU와 동일 항목 검증

#### MQTT 사용 시

- [ ] **브로커 주소**가 명시되었는가
- [ ] **포트 번호**가 명시되었는가 (기본 1883)
- [ ] **Subscribe 토픽** 목록이 있는가 (토픽명 + 페이로드 형식)
- [ ] **Publish 토픽** 목록이 있는가 (토픽명 + 페이로드 형식)
- [ ] 페이로드가 JSON인 경우 **필드 구조**가 명시되었는가

#### CNet (LS Electric) 사용 시

- [ ] **포트명/보드레이트**가 명시되었는가
- [ ] **디바이스 주소**가 LS 주소체계(%MW, %MB 등)로 작성되었는가
- [ ] **읽기 영역** (시작 주소 + 길이)이 명시되었는가
- [ ] **쓰기 주소**가 명시되었는가

#### MC (Mitsubishi) 사용 시

- [ ] **IP/포트**가 명시되었는가
- [ ] **디바이스 주소**가 Mitsubishi 주소체계(D0, M0 등)로 작성되었는가
- [ ] **읽기 영역** (시작 주소 + 길이)이 명시되었는가
- [ ] **쓰기 주소**가 명시되었는가

> **Why:** 통신 파라미터가 하나라도 빠지면 DeviceManager 코드를 정확히 작성할 수 없다. 특히 레지스터 맵 누락은 가장 빈번한 병목이며, 잘못된 주소로 쓰기를 실행하면 산업 장비 오동작으로 이어질 수 있다.

## 3. 통신

### Modbus RTU (해당 없으면 섹션 삭제)

- **포트명**: COM1 / /dev/ttyUSB0 등
- **보드레이트**: 9600 / 19200 / 115200
- **패리티**: None / Even / Odd
- **타임아웃(ms)**: 1000

**슬레이브 구성**

| 슬레이브 번호 | 역할 |
|-------------|------|
| 1 | 메인 컨트롤러 |

**레지스터 맵 (워드)**

| 주소 | 명칭 | 설명 | 단위 | R/W | FC |
|------|------|------|------|-----|-----|
| 0x0000 | RunState | 운전 상태 (0=정지, 1=운전) | - | R | FC3 |
| 0x0001 | ErrorCode | 에러 코드 | - | R | FC3 |
| 0x000A | Temperature | 온도 | 0.1℃ | R | FC3 |
| 0x0064 | SetPoint | 설정값 | - | R/W | FC3/FC6 |

**비트 맵 (해당 시)**

| 주소 | 명칭 | 설명 | R/W | FC |
|------|------|------|-----|-----|
| 0x0000 | RunCmd | 운전 명령 | R/W | FC1/FC5 |

### Modbus TCP (해당 없으면 섹션 삭제)

- **포트**: 502
- **타임아웃(ms)**: 1000
